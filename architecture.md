# Architecture.md — Логическая архитектура генератора сайта

## Назначение
Определить многоуровневую абстракцию для превращения контента (тексты, изображения, данные) в готовый статический HTML-сайт.  
Каждый уровень знает только о нижележащих, но не о вышележащих. Зависимости направлены строго вниз.

---

## Уровень 0: Ассеты и источники контента
**Ответственность:** Хранение сырых данных без поведения.

Содержит:
- Markdown-файлы с текстовым контентом
- JSON/YAML-файлы с метаданными и конфигурациями
- Изображения, шрифты, статические файлы
- Шаблоны Jinja2 (как ресурсы)

Не знает ни о каких других уровнях. Только предоставляет файлы для чтения.

---

## Уровень 1: Чистый HTML (Jinja2)
**Ответственность:** Превращение данных в HTML-строки по шаблонам.

Шаблоны Jinja2 получают переменные и возвращают HTML-фрагменты.  
Не знают о структуре страниц, не имеют логики приложения — только визуализация.

---

## Уровень 2: Python-контейнеры структуры элементов страницы
**Ответственность:** Хранить семантические элементы контента (заголовок, абзац, изображение, код) и уметь превращать себя в HTML-фрагмент.

Классы:
- `Section`
- `Paragraph`
- `Image`
- `CodeBlock`
- `List`
- `Quote`
- … и другие атомарные части страницы.

Зависимости:
- Знает о Mistune (для Markdown → HTML внутри текстовых элементов)
- Знает о Jinja2 (может использовать шаблон для своего рендеринга, если нужно)
- Не знает, как элементы собираются в страницу

Каждый элемент реализует метод `render(context) → str`, возвращающий HTML-фрагмент.

---

## Уровень 3: Python-узлы сборки страницы
**Ответственность:** Композиция элементов в логическую структуру страницы.

Узлы:
- `Page` — контейнер верхнего уровня, содержит метаданные (title, description) и список секций
- `Section` (как композит) — может содержать вложенные элементы и/или другие секции
- `Layout` (опционально) — определяет зоны страницы (header, content, footer)

Узлы хранят иерархию, но не рендерят себя полностью — только делегируют рендеринг элементам.  
Здесь же выполняется fail-fast валидация структуры (можно ли вложить Image в Page без Section).

Зависимости:
- Знает о контейнерах (уровень 2)
- Не знает о шаблонах целых страниц или инъекторе контента

---

## Уровень 4: Инъектор контента в дерево узлов
**Ответственность:** Наполнить дерево узлов конкретным содержимым, взятым из источников (уровень 0).

Получает на вход:
- Декларативное описание страницы (JSON/YAML) или напрямую Python-структуру
- Ссылки на Markdown-файлы, пути к изображениям и т.д.

Создаёт элементы (уровень 2) и вставляет их в узлы (уровень 3) согласно описанию.  
Применяет абстрактную фабрику для возможности замены классов элементов (темы).

Зависимости:
- Знает об узлах (3)
- Знает о контейнерах (2)
- Знает о файловой системе (0) — читает файлы
- Не знает о том, как будет происходить финальная сборка HTML-страницы

---

## Уровень 5: Сборщик страницы по имеющимся узлам
**Ответственность:** Рекурсивно обойти готовое дерево узлов и получить итоговый HTML-документ.

Вызывает `render(context)` у корневого узла.  
Не содержит логики рендеринга конкретных элементов — только управляет обходом и передаёт контекст.

Зависимости:
- Знает об узлах (3) и контейнерах (2)
- Не знает о том, откуда взялся контент (инъектор) и где хранятся шаблоны (Jinja2)

---

## Уровень 6: Пайплайн запуска сборщика
**Ответственность:** Оркестровать весь процесс от конфига до готового сайта.

Шаги пайплайна внутри этого уровня:
1. Прочитать конфигурационный файл (dev.yaml / prod.yaml)
2. Для каждой целевой страницы вызвать инъектор (4) с соответствующим описанием
3. Вызвать сборщик (5) для каждого полученного дерева
4. Сохранить готовые HTML-файлы в output/
5. (опционально) Скопировать статические ресурсы
6. (опционально) Запустить локальный сервер для предпросмотра

Зависимости:
- Знает об инъекторе (4) и сборщике (5)
- Не знает о конкретной структуре страниц (3) и деталях рендеринга элементов

---

## Вспомогательный слой: Контекст сборки
Пронизывает уровни 2–6.  
Это объект, который содержит глобальные параметры сборки:
- base_url
- пути к ассетам
- фабрику (тему) для создания элементов
- функцию resolve(md_path) для разрешения ссылок на контент

Передаётся во все вызовы `render(context)`.  
Сам контекст пассивен — не принимает решений, только предоставляет данные.

---

## Направление зависимостей

0 (файлы) → 1 (Jinja2) → 2 (элементы) → 3 (узлы) → 4 (инъектор) → 5 (сборщик) → 6 (пайплайн)

Каждый уровень зависит от всех нижних, но ничего не знает о верхних.

---

## Пример потока создания страницы

1. Пайплайн (6) берёт `page_home.json` и конфиг `dev.yaml`.
2. Инъектор (4) читает описание, загружает Markdown-файлы через уровень 0, создаёт элементы (2) через фабрику и вставляет их в Page-узел (3).
3. Сборщик (5) обходит Page, вызывает `render(context)` у каждой Section, Paragraph, Image.  
   Внутри Paragraph использует Mistune (через уровень 2) для Markdown → HTML.
4. Сборщик оборачивает результат в Layout через шаблон Jinja2 (уровень 1).
5. Пайплайн сохраняет файл в `output/`.

---

## Ключевые принципы, заложенные в архитектуру

- **Fail-fast:** проверка структурной целостности на уровне узлов (3) до рендеринга.
- **Абстрактная фабрика (темы):** позволяет переключать набор классов элементов (или шаблонов) без изменения инъектора или пайплайна.
- **Контент отделён от представления:** Markdown-файлы и JSON лежат на уровне 0, шаблоны Jinja2 на уровне 1, логика структуры на уровнях 2–3.
- **Изолированные компоненты:** любой уровень можно заменить, не трогая остальные, если сохранить интерфейс.
- **Понятный API для команды:** работа с генератором сводится к наполнению Markdown и JSON, а не к изучению внутренностей рендеринга.
